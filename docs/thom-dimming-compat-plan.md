Fix `docTypeForField()`: `/dim/` → `{ docType: "dimming_report", label: "Dimming Compatibility Report" }` (today it says "Dimensional Report" — wrong, see audit). The `dim_report`→`dimming_report` docType literal rename is safe — no other consumer exists (only the comment at `saleslayer.ts:838` and the label at `:855`, both being corrected; DC13). Correct the `docs/thom-bot-deferred-sources.md:20` row in the same PR. **Capture writes `kb_documents` + `product_documents` only — NO `dimming_report_products` rows at capture time** (DC4): reports (units) do not exist until extraction runs, so capture-time `field` links to `report_id` are impossible; all report↔product linkage is written at extraction time (§C.2).
2. **Never chunk/embed — atomic with the capture PR (DC7):** `kb_documents` rows with `doc_type = 'dimming_report'` are excluded from the docs-ingest Step B chunk/embed loop via SQL-level `.neq('doc_type', 'dimming_report')` in **BOTH** the `processPending` pending select (`apps/docs-ingest/src/index.ts:671` — verified: today it selects ALL `status='pending_extract'` rows with no doc_type filter, and `docs-ingest.yml` runs nightly on cron) **AND** the `--retry-failed` requeue (also unfiltered today). This exclusion + a test **ship IN the capture PR** (named in §G.2) — landing capture without it means the next nightly run text-extracts and embeds the mangled tables, precisely the RAG poison this plan exists to prevent. Terminal status: `--dimming` flips the kb row to **`active`** once its units are extracted — the `thom_doc_status` enum (`pending_extract|active|superseded|failed`, migration 0043) needs no new value; active-with-zero-chunks is invisible to `search_docs` by construction (it reads `kb_chunks`). No row sits `pending_extract` forever polluting `--dry-run` counts. The `product_documents` fan-out still gives every referencing SKU a "Dimming Compatibility Report" download button for free.
3. **Zip handling (docs-ingest, new step `--dimming`):** fetch bytes; `PK\x03\x04` magic → unzip in-process (`fflate`, tiny and sync — docs-ingest is a Node CLI with real RAM); iterate entries, keep `*.pdf` (case-insensitive), skip-and-log everything else (DWG etc.). Each PDF entry becomes one **extraction unit** identified by `sha256(entry bytes)` — idempotency is per-unit content hash, so a re-shipped zip with one changed member re-extracts only that member. **Byte-identical entries dedupe to one unit** with all their paths recorded in `zip_entry_path text[]` (DC4 — the Tube_Cube zip really contains duplicate entries under different folder names). Loose PDFs are a single unit (hash of the file). A changed Sales Layer file already produces a fresh `pending_extract` kb row via the existing hash-keyed capture — the unit hashes make re-runs incremental below that; the supersession sweep (§B.5) retires the old units.
4. **Fallback URL derivation:** for products where `zmatdimrep` is set (≠ `N/A`/`"0"`) but the `dim_report` FILE field is empty, derive `https://waclighting.com/storage/waclighting-images/dim_report/<urlencoded filename>` (verified pattern), HEAD-probe (MF spec-sheet HEAD-probe idiom, PR #208), and insert the same way with `source_system = 'derived_url'`. Failures logged, never fatal.
5. **Admin upload path for the rest of "a lot of files":** Davis's phrasing implies files beyond the PIM (rep-desk master charts, SharePoint). ## B. Structured extraction — Claude vision → strict rows, per-file verification gate

The engine is `apps/docs-ingest` (has Claude creds, R2 access, the CLI/Actions harness). New module `dimming.ts`, run `--dimming [--sample N]`.

1. **Extraction call:** per PDF unit, one Claude call with the PDF as a `document` block and a **forced tool call** (`tool_choice`) whose input schema IS the output contract — not free text like `claudeVision()`. Model: `claude-sonnet-4-5` default, env-overridable; the model id used is recorded per unit (DC11). Schema (zod twin in `packages/shared/src/thom/dimming.ts`, pure + tested):
   - report: `report_code` (**nullable** — 2018-era PDF bodies carry only "ID V0"; when absent, derive from the zip entry filename and mark the derivation, DC9), `product_family`, `skus_tested[]`, `related_model_patterns[]` (verbatim wildcards, e.g. `DS-CD05-*`), `test_voltage_range`, `control_types[]`, `test_notes` (**captures the 2025-era firmware line** — "CPU SCM: 03.56 …" — for VENTRIX the chart's validity is firmware-scoped, DC12).
   - rows: `manufacturer`, `dimmer_series` ("Diva", "Maestro"), `dimmer_model` (verbatim, parenthetical included), **`mode_qualifier` enum (`elv | triac | null`)** parsed from the parenthetical in the model/series cell (DC2 — `Adorne Touch (ELV)` vs `(TRIAC)` both sit under the Adaptive section; the qualifier is per-row information, NOT section membership), `related_dimmer_models[]` (2025-era layout), `phase_type` enum (`adaptive | elv | triac | zero_to_ten_v | other` — from the SECTION the row sits under; `other` absorbs the "iOS or Android APP" section, DC12), `test_voltage` (**text**, not numeric — the WAC MyWAC app row reads `120-277`, DC12), `low_end_pct` (number | null), `comments` (verbatim).
   - **`status` is derived in code, not by the model (DC1 — reworked):** four-value enum, conservative by default.
     - `not_recommended` iff comments contain "Not Recommended".
     - `not_compatible` iff `low_end_pct` is null AND comments match the non-function list ("not responsive", "does not find appropriate load", "phase control not supported").
     - **`tested_issue`** (fourth, conservative status) for ANY row with `low_end_pct = null` that matches neither of the above — **a null low end can NEVER derive `tested_compatible`** (hard rule; the real charts carry `HRD-5NE, 120V, N/A, "Flashing at all dimming levels"` rows that v1 would have published as compatible). Rendered as "tested with issues, see the chart's comment" + the verbatim comment.
     - `tested_compatible` only when `low_end_pct` is numeric and no "Not Recommended" note (blank comment + numeric low-end = compatible-as-tested is faithful to the chart; we still never invent the words "Compatible"/"Recommended").
     - **Unmatched-vocabulary human review:** the extraction run collects the distinct comment strings that drove a `tested_issue`/`not_compatible` classification outside the known list and prints them in the run report; **Davis reviews the vocabulary before rows activate** (extends the §G.3 gate). Pure function, table-tested against every audited comment string including the DC1 finds.
   - Normalization in code: **strip the `mode_qualifier` parenthetical FIRST** (before any key building — otherwise "collapse spaces" fuses `ADTP703TU (ELV)` into garbage, DC2), then U+2010/U+2011 → `-`, collapse spaces inside model codes, uppercase for the match key (original preserved for display). **Slash-expansion for related-model shorthand (DC8):** `AYCL-153P/253P` → `AYCL-153P` + `AYCL-253P` (shared prefix + alternate suffix); `HQRD-6ND/10D` → `HQRD-6ND` + `HQRD-10D`; bare `-` placeholder cells are dropped, never become a model. The same U+2010→`-` + uppercase normalization applies to `related_model_likes` pattern building (the text layer's patterns carry U+2010: `DS‐CD05‐*`).
2. **Per-file verification gate (the R12 spot-check, mechanized — reworked per DC5):** after extraction, issue an independent second Claude call per file with **stratified sampling: at least one row per section per page** (random N=3 systematically under-samples page-2 bands — "Forward Phase Dimmers (TRIAC) Cont." and "0-10V Dimmers" start mid-page-2 in both Tube/Cube units). The verification question **includes the mode qualifier** and asks (a) the measured low-end % and comment for "model X *(qualifier)* at voltage V" and (b) **"under which section header does this row appear"** — compared against `phase_type`, the one field the plan itself identifies as the catastrophic failure mode. Add a **per-section row-count cross-check** question. Rows whose (model, voltage) key is duplicated in the unit are skipped or disambiguated by qualifier — never asked ambiguously ("ADTH700RMTUM1 at 120V" matches TWO rows, 9.0% and 21.0%, in the 2018 layout; an unqualified question is a coin flip). **Verifier model: `claude-haiku-*` by default** (cheaper and more independent than "different framing" on the same model; env-overridable). Any mismatch → the unit is marked `needs_review` and its rows stay inactive; matches → `active` + `verified_at`. The `--sample` run prints a human-readable verification report (extracted row count, checked cells, per-file pass/fail, **the unmatched comment vocabulary from B.1**) for Davis's eyeball pass.
3. **Product↔unit binding — written at extraction time, pattern-primary (DC4, adjudicated):** ALL `dimming_report_products` rows are written by the extraction run, never at capture. `pattern` matching (unit's in-PDF `related_model_patterns`/`skus_tested` against `products.sku` + variant SKUs) is the **primary** binding — the zip's own folder names and filename wattages are provably untrustworthy (byte-identical entries under contradictory names). `link_kind='field'` is permitted ONLY when the source file is a single loose PDF (file identity = unit identity); a multi-unit zip NEVER fans a file-level product link to its units. A product that references a zip but matches no unit's patterns gets a **family-level answer** ("WAC Group's tested charts for this family cover these sizes/wattages: …") — never an arbitrarily chosen unit. **Build-time overlap audit:** any SKU matched by patterns from 2+ units with different `product_family` values → flagged in the run report, its link rows held for review. Links are **rewritten per product on each run** (stale links from a moved `zmatdimrep` die with the sweep, §B.5).
4. **Extraction versioning:** `DIMMING_EXTRACTION_VERSION` const in shared. Rows carry it plus the model id; `--dimming` re-processes any active unit whose stored version < current (schema or prompt fix ⇒ bump ⇒ targeted re-run). Per-unit rewrite is transactional delete-then-insert keyed by the unit, so a failed re-extract never leaves a half-report.
5. **Supersession sweep (DC6 — new):** every `--dimming` run ends with a sweep: `dimming_reports` whose source `kb_document`/content hash is **no longer referenced** by any current product field value, derived URL, or admin upload → `status = 'superseded'`, excluded from tools. Without this, a changed chart file (new hash → new kb row → new units) leaves the OLD units active forever and both V0 and V1 answer simultaneously (`captureDocs` explicitly defers link pruning — see the docstring in `apps/api/src/saleslayer.ts`; §H's v1 claim that hash-change re-queue "covers the realistic drift" covered re-extraction of the NEW file only). Product→report links are rewritten per product on each run (B.3).
6. **Failure posture:** per-unit try/catch; a bad PDF marks that unit `failed` with `last_error` and never blocks the batch (ingest CLI idiom).

## C. Schema — migration `0064_thom_dimming.sql`

Three tables, all the 0048/0061 posture: **no FK to products** (prunable denormalized cache keyed by text; DC15 — counter-planner verified 0048 matches this no-FK/service-write posture), **open read** (`for select using (true)` — every row derives from files publicly linked on brand PDPs; no internal rows; do not reintroduce the bare per-row `is_active_*()` 0055 incident form), **service-role-only writes** (no write policy), covered by the anon-boundary test. Under open-read RLS anon can read `needs_review`/`superseded` rows — which is why the tools filter on status explicitly (§D, DC14).

1. **`dimming_reports`** — one row per extraction unit: `id`, `kb_document_id uuid` (provenance, no FK), `source_url`, **`zip_entry_path text[] null`** (DC4 — byte-identical deduped entries keep ALL their paths; alternatively first-seen + log, but the array is chosen), `content_hash text unique` (idempotency key), **`report_code text null`** (DC9 — absent from 2018-era PDF bodies; filename-derived when null, flagged as derived), `product_family`, `skus_tested text[]`, `related_model_patterns text[]` (verbatim) + `related_model_likes text[]` (normalized: U+2010→`-`, uppercase, `*`→`%`), `control_types text[]`, `test_voltage_range text`, `test_notes` (incl. firmware line), `extraction_version int`, **`model text`** (extracting model id, DC11), `status text check (status in ('pending','active','needs_review','failed','superseded'))` (DC6 adds `superseded`), `verified_at`, `last_error`, `synced_at`, `created_at`.
2. **`dimming_report_products`** — report ↔ catalog linkage, `(report_id, product_sku, link_kind)` unique; `link_kind check in ('field','pattern')`. **ALL rows written at extraction time** (DC4): `pattern` rows resolved from `related_model_likes`/`skus_tested` against `products.sku` + variant SKUs (primary binding); `field` rows ONLY for loose single-PDF sources where file identity = unit identity. Rewritten per product each run. Index on `product_sku`.
3. **`dimming_compat_rows`** — `report_id`, `manufacturer`, `dimmer_series`, `dimmer_model` (display, parenthetical intact), **`mode_qualifier text null check (mode_qualifier in ('elv','triac'))`** (DC2), `dimmer_model_norm` (match key — qualifier stripped, U+2010 fixed, uppercased), `related_dimmer_models text[]`, `related_dimmer_models_norm text[]` (slash-expanded, placeholder-dropped, DC8), `phase_type check in ('adaptive','elv','triac','zero_to_ten_v','other')` (section-derived, kept BESIDE the qualifier), **`test_voltage text`** (DC12 — `120-277` exists), `low_end_pct numeric null`, **`status check in ('tested_compatible','not_recommended','not_compatible','tested_issue')`** (DC1 adds the fourth conservative status), `comments`, `extraction_version`. Indexes: `(report_id)`, `(dimmer_model_norm)`.
4. **No pg_trgm** (not installed per audit; counter-planner could not cheaply re-verify but agrees the TS-side decision is right at ~10⁴ rows regardless, DC15) — fuzzy matching is TS-side over a broad SQL prefilter (§D.3): a manufacturer-or-prefix `ilike` prefilter plus pure-TS ranking is simpler, testable in shared, and adds no extension surface.

Verify block: RLS as-anon PASS/DENY checklist (0052 idiom), the unique-hash conflict test, and recorded row counts from the sample run.

## D. Tools — forward and reverse, both surfaces

New module `packages/shared/src/thom/dimmingTools.ts`, composed behind flag `THOM_DIMMING` (spec-rank dark-launch idiom: `env.ts:41` member, `agent.ts:62` composition pattern, committed `vars` in both `apps/api/wrangler.jsonc` + `apps/thom-bot/wrangler.jsonc` — internal first, public second; all verified to exist, DC15). Both tool names join `PUBLIC_TOOL_NAMES` (`tools.ts:1055`) in the same PR (allow-listing is inert until the flag advertises them). **Both tools filter `dimming_reports.status = 'active'` explicitly in the join** (DC14 — under open-read RLS "rows stay inactive" is only true if the query says so; only the report carries status) — stated here, tested in §F.

1. **`check_dimmer_compatibility`** (forward): `{ product: string, dimmer?: string }`. Resolve product → **active** reports via `dimming_report_products` (direct SKU, variant-SKU→parent via the products map, then `related_model_likes` match). No unit match but the product's family has tested units → the **family-level answer** (DC4): "WAC Group's tested dimming charts for this family cover these sizes/wattages: …" — never an arbitrarily chosen unit. No report at all → the honest-miss line (§E). With `dimmer`: fuzzy-match rows (D.3) and answer per phase type AND mode qualifier — the same physical dimmer can be `tested_compatible` as (ELV) and `not_recommended` as (TRIAC) **within the same Adaptive section** (DC2: Adorne Touch, ADTP703TU, MA-PRO, WNRL50), so the answer MUST name the section-derived phase type plus the row's mode qualifier when present. Without `dimmer`: summarize the report — counts per phase type, the tested-compatible list capped at ~30 lines with "+N more" (MAX_ACCESSORY_LINES idiom), always low-end % + verbatim comment per named row; `tested_issue` rows always carry their verbatim comment. **Every answer prints the report's own `skus_tested` + `product_family`** (DC4 — scope errors must be visible to the user), cites `report_code` (or "chart, ID from filename" when derived) + links the source PDF (the product_documents button already exists), and carries the chart's own caveat line (single-fixture test; check the dimmer's load rating for fixture counts per dimmer).
2. **`find_products_for_dimmer`** (reverse): `{ dimmer: string, status?: 'any'|'tested_compatible' }`. Normalize → match `dimming_compat_rows` (including `related_dimmer_models_norm` — a 2025-era report tested `MA-PRO` "for" `RRD-PRO` too; slash-expanded forms match, DC8) → group by **active** report → resolve to product families / SKU patterns via `dimming_report_products` + `related_model_patterns`.
